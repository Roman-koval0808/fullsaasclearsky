Profile Creation — What Actually Happens at Form Submit
The Six-Step Merge Sequence
The hub profile does not exist until this sequence completes. Every step is a discrete operation. Any step can fail independently. The sequence is the same whether the visitor is new or has converted before.
01
Conversion detected — pixel fires immediately
The form submit event fires before anything else. form_submitted logged to ClearSky event stream. Score delta +20 applied. Conversion moment flag set. This step cannot fail — it is a client-side write only.
{ event: "form_submitted", sessionId: "cs_b22f…7a91", bucket: "active", delta: 20, conversionMoment: true }
PENDING
02
Contact details extracted and hashed client-side
Name, phone, and email read from the form. Phone normalised to E.164 format. Email lowercased and trimmed. Both SHA-256 hashed in the browser before transmission. PII never travels in plaintext. Hash convention must match hub team exactly — a mismatch creates silent duplicate profiles.
name: "Marie Tremblay" (plaintext — display only)
emailHash: sha256("m.tremblay@gmail.com") → "a7f3c…"
phoneHash: sha256("+17052740988") → "9b2e1…"
PENDING
03
Hub resolve call — does this person already exist?
GET /hub/profiles/resolve called with hashed email and phone. Three possible responses. Single match: existing profile found — proceed to stitch. No match: new person — create profile. Multiple matches: conflict — hold, flag, escalate to support. Never auto-merge on conflict.
GET /hub/profiles/resolve?email=a7f3c…&phone=9b2e1…

→ Response: { "profileId": null, "isNew": true }
No existing match — new profile will be created
PENDING
04
Profile created + session history attached
POST /hub/profiles/merge creates hub_rf_0091. Complete anonymous session history from this browser sent in the payload — all four iPhone sessions via cookie continuity plus this laptop session. Hub appends history in chronological order, interleaved with any prior communication history. Profile is now Tier 1.
POST /hub/profiles/merge
{ profileId: "hub_rf_0091", anonymousId: "cs_b22f…7a91",
sessionHistory: [ session_1, session_2, session_3, session_4 ],
contactDetails: { emailHash: "a7f3c…", phoneHash: "9b2e1…", nameProvided: "Marie" }}

→ Response: { "merged": true, "profileId": "hub_rf_0091" }
PENDING
05
Cookie updated — anonymous ID retired
The cs_b22f…7a91 anonymous cookie is updated to carry hub_rf_0091. Any future session from this browser is immediately Tier 1 — no form needed. Anonymous records retained 30 days for audit trail then expired per tiered schedule. The anonymous ID is never deleted immediately — it is the audit chain.
Cookie before: { anonymousId: "cs_b22f…7a91", hubProfileId: null }
Cookie after: { anonymousId: "cs_b22f…7a91", hubProfileId: "hub_rf_0091" }

Next visit from this browser → Tier 1 from first pageview

What happens at each possible outcome
✓
Husband submits form with his own contact details
Hub resolve call fires with his phone and email hashes. No match found in hub. New profile created — hub_rf_0092. Co-occurrence candidate between his laptop and Marie's iPhone evaluated: his form data does not match Marie's hub record. Candidate link discarded. Two separate profiles. No merge. No false positive.
→
Neither person submits a form that evening
Both sessions expire independently per bucket grace period. Co-occurrence flag expires with the 30-minute window. Nothing is stitched. Nothing is merged. The husband's session is a fresh anonymous visit. Marie's iPhone sessions remain exactly as they were — linked to her cookie and fingerprint, unaffected.
⚠
Husband submits form with Marie's contact details
Hub resolve call finds Marie's existing profile hub_rf_0091. System flags a merge conflict — existing profile plus new session from different fingerprint. Does not auto-merge. Holds the stitch and escalates to the rep for manual confirmation. Rep sees both sessions and makes the decision. This is the one edge case that requires human review.
✗
What the system will never do
Auto-merge two profiles based on IP co-occurrence alone. Transfer Marie's score or history to the husband's session. Treat the 62% co-occurrence flag as a confirmed identity match. Take any customer-facing action on the husband's session beyond serving the website — no A2P, no cross-channel contact, no push engine personalisation beyond the Orchestrator's normal bucket-based content decisions.
The architectural principle that prevents false positives
The co-occurrence flag is a candidate only. It never triggers a stitch. It never transfers history. It never upgrades identity. It sits in the session store and waits. Conversion data is the only arbiter of identity. A hub resolve call with matching contact details is the only event that can stitch sessions together. The IP match tells the system to watch — it does not tell the system to act.
